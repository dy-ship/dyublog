亚马逊利润计算：从 Excel 到 Python 的完整迁移

把售价、头程、FBA、退货率、广告费率、佣金统一进一个可维护的脚本，并导出 Excel 模板供下载。

在很多卖家刚起步的时候，通常会依赖 Excel 来计算利润。通过设置公式、引用单元格，快速得出一个大概的毛利率和净利率。然而，当业务逐渐扩张、SKU 越来越多时，Excel 就会变得臃肿且难以维护，尤其是涉及退货率、广告费、促销折扣等因素时，表格往往要复制多份才能覆盖各种场景。

于是，我尝试用 Python 来重构这一套计算逻辑。Python 的优势在于可以清晰地定义每个参数：售价、头程运费、FBA 配送费、亚马逊佣金、广告费率、退货率等。每个参数都由用户手动输入，不再依赖繁琐的单元格引用。代码会输出完整的利润结果，包括毛利润、毛利率、净利润、净利率，并且可以自动导出到 Excel 文件，既保留了可读性，又方便后续数据整理。

更重要的是，迁移到 Python 后，逻辑可以随时扩展。比如，当你想增加「退货不退佣金」这样的特殊规则时，只需要在函数里加一行代码即可。而在 Excel 中，往往需要修改几十个单元格，风险更大。最终效果是：计算更稳定，复用更简单，也为未来做自动化铺平了道路。

个人网站 7 天上线：从草稿到可评论的博客

部署、评论系统、SEO 最小可行集与常见坑位清单。

很多人想写博客，却常常卡在第一步：用什么工具搭建？是用 WordPress，还是 Hexo、Hugo 这样的静态博客框架？其实，如果目标是「尽快上线」，完全可以用最简单的 HTML + CSS 架构，7 天时间就能跑通一个可用的版本。

我的流程是这样的：第一天写一个最小的首页框架，包括导航栏和几篇文章的占位符；第二天到第三天，整理内容区块，比如精选文章、项目展示、工具分享等，让页面层次清晰；第四天，加入评论系统，我选择了 Disqus，只需注册账号、拷贝一段代码，就能在文章下方启用评论；第五天，处理 SEO 基础项，例如 meta description、title、keywords，确保页面能被搜索引擎收录；第六天，把代码推到 GitHub，再通过 Zeabur 部署，一键生成访问链接；第七天，做一些微调，包括移动端适配、字体和配色的统一。

结果就是：一周时间，一个能发文章、能互动、能被搜索的个人网站上线了。相比纠结用什么框架，不如先做一个最小可行产品（MVP），之后再慢慢升级迭代。

产品图可复用工作流：PS 图层命名与导出规范

一套命名 + 导出规范，减少返工并提升上图效率。

在亚马逊上，产品图片的重要性不言而喻。但很多卖家在处理图片时，往往只顾着「做出来」，却忽视了规范性。比如，图层随意命名为“图层1”、“图层复制”，时间一久，文件变得难以理解，别人接手时几乎无法继续工作。

为了解决这个问题，我总结了一套可复用的工作流。首先是图层命名：每个图层都要有明确含义，比如「背景_white」、「主机_body」、「电池_battery」、「文字_tagline」。这样即使过了几个月，再次打开文件也能立刻定位到目标元素。其次是分组，把相似元素放进文件夹，如「配件」、「LOGO」、「光效」，保持层级清晰。

导出规范同样重要。比如主图必须 1000x1000 白底，细节图可以 1500x1500 并带局部放大效果，色卡图要统一背景和尺寸。文件命名上，也应该遵循「SKU_用途_尺寸」的规则，比如「mower_red_main_1000.jpg」，避免混乱。

有了这一套规范，不仅效率更高，还能减少返工。更重要的是，规范是可以移植和复用的。你在一个产品上建立了标准，后续所有产品都能直接套用，团队协作也更顺畅。
